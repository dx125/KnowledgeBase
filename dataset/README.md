# `dataset/` — the authoritative dataset definition

This folder is the **single source of truth for how the knowledge-base dataset is
built, versioned and restored.** If the live database is lost, everything needed to
recreate the resulting dataset — without losing the context of what was changed, why,
and what mistakes were made — lives here plus the committed patch layers.

It exists because the dataset is not one file: it is a **read-only raw vendor dataset**
replayed through **git-owned patch layers** into Supabase Postgres. This folder
documents that pipeline and pins its versions, schemas and provenance.

## Files

| File | What it is |
|---|---|
| [`MANIFEST.json`](MANIFEST.json) | Machine-readable description of the resulting dataset: raw version it derives from, each patch layer with a content hash + counts, the schema migrations, and the build pipeline. Regenerated by `scripts/build-manifest.mjs`. |
| [`SCHEMA.md`](SCHEMA.md) | The complete schema: DB tables (Postgres), every patch-layer file shape, the upstream evidence model (claims/clean_messages), the Q&A model (questions→cards), and the versioning/metadata fields. |
| [`CHANGELOG.md`](CHANGELOG.md) | The full history — upstream vendor releases (v5.9 → v6.6) **and** every change we made after (v6.6 de-collapse → v7.3), each with what / why. |
| [`PROVENANCE.md`](PROVENANCE.md) | Why decisions were made, the **mistakes and issues we hit and how we fixed them**, the anti-loss evidence model, security caveats, and how to regenerate each layer. |
| [`schemas/`](schemas/) | JSON Schemas for our patch layers + copies of the upstream file schemas, for self-contained validation. |

The raw, per-card editorial corrections themselves live one level up in
[`../dataset-patches/`](../dataset-patches/) (the layers); this folder is the *map* of them.

## The pipeline (how the resulting dataset is built)

```
 Telegram chat export (messages.jsonl, 145k msgs, 2023-02 … 2026-06)
        │   vendor pipeline (upstream "first dataset developer")
        ▼
 clean_messages.jsonl ──► claims.jsonl ──► cards (kb_cards.json) ──► locale_{ru,en,es,de}.json
        │   anti-loss evidence layer: every card traces back to source message IDs
        ▼
 RAW VENDOR DATASET  (DATASET_DIR, currently kb_dataset_uy v6.6.0-decollapsed)   ← read-only
        │
        │   our git-owned patch layers (re-applied on every deploy):
        ├── dataset-patches/card-overrides.json  EN/ES/DE corrections of collapsed vendor text
        ├── dataset-patches/new-cards.json        editorial cards the vendor never produced
        ├── dataset-patches/faq.json              Q&A answer-cards (topic.faq_* topics)
        └── dataset-patches/questions.json        questions → answer-cards (+ ask_frequency)
        │
        ▼   scripts/deploy.mjs  (one atomic transaction: delete-all + insert-all + version row)
 Supabase Postgres  ──►  Edge Function API (supabase/functions/kb)  ──►  web app + mobile
```

Key property: the raw files on disk are **never edited**. Every change we own is a
patch layer, so it **survives a new raw-data drop** — re-applied deterministically on
each deploy. The deploy is a full atomic replace, so it also handles edits, renames and
deletions; nothing stale lingers.

## How to UPDATE the dataset (a new raw vendor drop arrives)

1. Unzip the new raw dataset; point `DATASET_DIR` (in `.env`) at it.
2. `cd scripts && node check-overrides.mjs` — lists RU-source drift (re-translate those)
   and any vanished card_ids (update/drop those override keys).
3. `node build-questions.mjs` — recompute `questions.json` ask_frequency against the
   current raw chat (optional; preserves values if the chat file is absent).
4. `node build-manifest.mjs` — refresh `dataset/MANIFEST.json`.
5. `node deploy.mjs --dry-run` to preview, then `node deploy.mjs --label "vX.Y …" --notes "…"`.
6. `supabase functions deploy kb` only if the Edge Function changed.
7. Append a `dataset/CHANGELOG.md` entry (what changed and why).

## How to RECREATE the dataset from scratch (DB lost)

1. Provision a fresh Supabase project; set `.env` (`DATABASE_URL`, `SUPABASE_*`, `DATASET_DIR`).
2. `cd scripts && node migrate.mjs` — apply all `supabase/migrations/*.sql` (schema + RPCs).
3. Restore the raw vendor dataset into `DATASET_DIR` (the v6.6 folder, or rebuild it with
   `node build-dataset-version.mjs` from an earlier raw + `card-overrides.json`).
4. `node deploy.mjs --label "restore"` — the patch layers are re-applied automatically.
5. `supabase functions deploy kb`; `npm --prefix ../web run build` for the web app.
6. Verify with `node smoke.mjs` and `GET /version`.

Nothing in this restore depends on un-committed state except the raw vendor dataset and
the project secrets. The raw dataset is itself reproducible — its schemas and manifest are
mirrored in [`schemas/upstream/`](schemas/upstream/) and summarized in `SCHEMA.md`.
